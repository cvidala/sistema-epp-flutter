  <name>Task 1: Crear .github/workflows/test.yml (CI-01, CI-02, CI-03, CI-04)</name>
  <files>.github/workflows/test.yml</files>
  <action>
    Crear el directorio .github/workflows/ si no existe y escribir test.yml con la siguiente estructura exacta:

    Triggers: push a main, pull_request targeting main.

    Job name: test. runs-on: ubuntu-latest.

    Pasos en orden:

    1. actions/checkout@v4

    2. subosito/flutter-action@v2 con flutter-version: '3.24.x' y channel: 'stable'. Usar cache: 'pub' para acelerar el pipeline.

    3. flutter pub get

    4. flutter analyze --no-fatal-infos — step name "Analyze". Si retorna código de salida distinto de 0, el job falla (CI-03). No usar --no-fatal-warnings para no enmascarar errores reales.

    5. flutter test test/unit/ test/widget_test.dart --coverage — step name "Unit & Widget Tests". El flag --coverage genera coverage/lcov.info. No requiere secrets (CI-01, CI-02).

    6. flutter test test/integration/supabase/ test/integration/e2e/ --tags integration — step name "Integration Tests". Requiere todos los secrets como variables de entorno:
       SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
       SUPABASE_ANON_KEY: ${{ secrets.SUPABASE_ANON_KEY }}
       SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.SUPABASE_SERVICE_ROLE_KEY }}
       TEST_ADMIN_EMAIL: ${{ secrets.TEST_ADMIN_EMAIL }}
       TEST_ADMIN_PASSWORD: ${{ secrets.TEST_ADMIN_PASSWORD }}
       TEST_SUPERVISOR_EMAIL: ${{ secrets.TEST_SUPERVISOR_EMAIL }}
       TEST_SUPERVISOR_PASSWORD: ${{ secrets.TEST_SUPERVISOR_PASSWORD }}
       TEST_READONLY_EMAIL: ${{ secrets.TEST_READONLY_EMAIL }}
       TEST_READONLY_PASSWORD: ${{ secrets.TEST_READONLY_PASSWORD }}
       TEST_OBRA_ID: ${{ secrets.TEST_OBRA_ID }}
       TEST_OBRA_ID_NO_ACCESS: ${{ secrets.TEST_OBRA_ID_NO_ACCESS }}

    7. actions/upload-artifact@v4 — step name "Upload Coverage". name: coverage-report. path: coverage/. Solo ejecutar si el job no fue cancelado: if: always(). Esto sube tanto lcov.info como el directorio html si se generó (CI-04).

    NO incluir integration_test/ (los tests E2E con Xcode están excluidos de CI per instrucciones).
    NO usar continue-on-error en ningún step — todos deben fallar el job limpiamente (CI-02).
    Usar timeout-minutes: 15 en el job como guardrail de emergencia (el target es < 5 min, 15 es el hard limit).
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → Supabase | Secrets expuestos como env vars en el runner efímero — mitigado por GitHub Encrypted Secrets |
| .env.test → git history | Credenciales de test reales que no deben commitearse jamás |
| PR de fork externo | Forks no tienen acceso a secrets en GitHub Actions por defecto — comportamiento correcto |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | .env.test | mitigate | .gitignore excluye .env.test; verificado en Task 4 con git ls-files |
| T-04-02 | Information Disclosure | GitHub Actions env vars | accept | GitHub Encrypted Secrets nunca se loguean en plaintext; runner efímero se destruye post-job |
| T-04-03 | Tampering | PR de fork sin secrets | accept | GitHub Actions no pasa secrets a forks por defecto; integration tests fallarán por falta de env vars, no por inyección |
| T-04-04 | Elevation of Privilege | SUPABASE_SERVICE_ROLE_KEY en CI | mitigate | Key de test, no producción; configurar en GitHub Secrets (nunca en workflow YAML); acceso de lectura/escritura solo a datos de test |
| T-04-SC | Tampering | subosito/flutter-action@v2 (tercero) | accept | Action establecida y ampliamente usada (>10M usos); pinned a v2 major tag; riesgo de supply chain bajo para CI tools |
</threat_model>
